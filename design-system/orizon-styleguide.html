01 · Fundamentos

Cores

Todos os componentes referenciam tokens — nunca hex literal. Alterne o tema acima e observe: os mesmos tokens resolvem para valores diferentes automaticamente.

⚑Regra v9: nenhum componente declara cor literal. Sempre var(--accent), var(--btn-primary-bg) etc. Foi exatamente um hex hardcoded que causou o bug do "+ Novo Projeto" com duas cores.

03 · Componentes

Botões

⚑Decisão v10: o botão primário quebra deliberadamente o padrão do --accent. No claro, cinza neutro (contraste tonal, sem matiz de marca). No escuro, fundo aquecido quase preto — o destaque vem da linha bronze 1.5px (--accent-line) + sombra forte, não do preenchimento (variação F aprovada). Máximo 1 primário por tela.

Tamanhos: sm 30px · padrão 38px · lg 44px. Secundário: borda visível no claro, só sombra no escuro (correção v5 — caso "Imprimir Orçamento"). Perigo: transparente com borda/texto --err, tint no hover.

04b · Componentes

Dimensões — alturas e larguras padronizadas

⚑Regras: na mesma linha, todo controle (campo ou botão) tem a mesma altura — sm 30 · padrão 38 · lg 44. Largura de campo é definida pelo tipo de dado (classes w-xs…w-full), nunca decidida por tela. Componente nunca recebe width/height inline.

Alinhamento em linha — campo, select e botão na mesma grade de 38px:

Largura pelo tipo de dado — o tamanho do campo comunica o que cabe nele:

Taxonomia por papel: Ação (muda dados → botão) · Entrada (coleta dados → campo) · Indicador (só informa → badge, alerta, KPI) · Navegação (move você → aba, sidebar, status selector). Cada família usa exclusivamente a sua escala.
